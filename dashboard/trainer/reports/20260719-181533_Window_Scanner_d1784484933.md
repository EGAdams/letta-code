## Wrapper defect diagnosed

1. **`check_duplicates` only matches exact (date, amount) pairs.** It cannot detect
   aggregate duplication — a summary/acknowledgment receipt whose total is the sum of
   several already-recorded smaller payments. Mazda's own document text ("acknowledging
   multiple prior payments") is exactly this case. This is a real gap in the duplicate
   tooling, not a missing save call.
2. Mazda's first proposal (89, "always run the save step ... do not leave stored=false
   when storage is available") was the **wrong lesson** — it would have removed the
   safeguard that correctly stopped a likely double-count, rather than adding the real
   missing check (aggregate/sum duplicate review against vendor history).
3. `judge_trace`'s deterministic rule (`failed: expense_stored`) treats every no-store
   outcome as FAIL with no room for "correctly declined to store" — worth a human look;
   this run being judged FAIL both before and after correction is a known judge
   limitation, not a Mazda defect.
