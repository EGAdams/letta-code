SSH into this machine

This box is DESKTOP-SHDBATI — a Windows 10 Home host running WSL Ubuntu 24.04. There are two separate SSH targets, each with its own user. Almost everything (Letta server, agents, /home/adamsl, the code) lives inside WSL Ubuntu — that's the one you usually want.

Always connect over Tailscale, never the LAN IP. The 10.0.0.x LAN address is DHCP and changes on reboot (it shifted 10.0.0.143 → 10.0.0.142 on 2026-05-30, which is part of why connections broke). Tailscale IPs never change. Both ends must be on the same Tailnet — run tailscale status to confirm.
TargetConnect (Tailscale — use this)UserLAN fallback (drifts!)
WSL Ubuntu
(Letta, agents, code)		 ssh adamsl@100.80.49.10 adamsl not on LAN directly
Windows 10 host ssh NewUser@100.69.80.89 NewUser ssh NewUser@10.0.0.142

Both run OpenSSH on port 22.

Why SSH broke on 2026-05-30 — and how it's fixed

After the Windows reboot at 17:00, two things went wrong:

  1. WSL's sshd did not auto-start. The ssh.service was set to disabled, so it stayed down from the 17:00 reboot until someone started it manually at 19:13 — anyone trying to connect in that window got "connection refused."
  2. The LAN IP drifted 10.0.0.143 → 10.0.0.142 via DHCP, so anyone using the old LAN address was hitting a dead host.
Permanent fix — run once in WSL so sshd always comes back after a reboot: 
sudo systemctl enable ssh
sudo systemctl start ssh
After that, always connect over Tailscale (adamsl@100.80.49.10 for WSL) — that address never changes, so reboots can't break it.

Troubleshooting checklist

"Connection refused" or "timed out" 
# On the WSL host, confirm sshd is up and set to auto-start:
systemctl is-active ssh        # want: active
systemctl is-enabled ssh       # want: enabled
ss -tlnp | grep :22            # want: 0.0.0.0:22 listening

# Confirm Tailscale is connected (run on BOTH ends):
tailscale status
tailscale ip -4                # WSL should report 100.80.49.10

# End-to-end reachability (proves the box answers over Tailscale):
curl -s http://100.80.49.10:8283/v1/health/   # want: {"status":"ok"}
