The bug (root cause, verified in letta-src)

For a git_enabled agent (has the git-memory-enabled tag), letta/schemas/memory.py Memory.compile() renders only blocks whose label starts with system/:

if self.git_enabled:
    return [b for b in self.blocks if b.label and b.label.startswith("system/")]

Proper enablement (block_manager_git.py::enable_git_memory_for_agent) relabels every block to system/<label>. But these agents were enabled the manual way (INSERT the tag + git init --bare) without relabeling → every block is filtered out → the agent runs with no persona / human / custom memory at all. Orphaned memfs repo files map to no block and never reach context either.

This was NOT the "duplicate team_agents block" the memory_system_plan.html assumed. That dedup is moot. The real disease is total memory non-render. Likely explains erratic history (Scissari reset-loops, etc.).

The fix procedure (proven, reversible)

  1. Confirm the agent's blocks aren't shared: each block_id has n_agents=1 in blocks_agents.
  2. Save a rollback snapshot of current labels.
  3. In a transaction, scoped to that agent's block ids, guard NOT LIKE 'system/%': 
    UPDATE block          SET label='system/'||label        WHERE id IN (...);
UPDATE blocks_agents  SET block_label='system/'||block_label WHERE agent_id=...;
-- also blocks_conversations.block_label if any rows
    Commit only if row counts == #blocks.
  4. POST /v1/agents/<id>/recompile?dry_run=true → verify <self>/<memory> + content present.
  5. POST /v1/agents/<id>/recompile (persist).

block has no unique constraint on label, so no collision risk. Label lives in block.label + blocks_agents.block_label (+ blocks_conversations.block_label).
