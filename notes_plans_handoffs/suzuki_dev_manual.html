Preface

This manual rewrites the Mazda developer document for Suzuki, a new Mazda-type Letta agent whose domain is software debugging instead of finance verification. Mazda proved the pattern: orchestrator, specialist minions, deterministic verifiers, wrapper revisions, experiments, gates, activation, and rollback. Suzuki keeps that machinery and swaps the domain harness from receipts and statements to repositories, failing tests, stack traces, patches, and regression checks.

The Central Thesis

All Mazda-type agents must share one self-improvement core. A new agent changes only the harness: task models, tools, evaluators, workflows, gates, reports, and minion templates. The kernel, evidence store, wrapper revision system, experiment runner, activation service, and rollback service stay common.

The practical goal is simple: Suzuki should find bugs faster and more reliably each time she debugs. Every run should leave evidence and a lesson: the failing command, the reproduction path, the diagnosis, the patch, the verification result, the regression result, and any approved wrapper improvement that will help the next run.

How to read this manual

Part I defines Suzuki and the proposed bug-hunting fleet. Part II describes the shared core, the harness boundary, the new debugging interfaces, and the self-improvement loop. Part III gives the migration plan and operating checklist.

Part OneFoundations

In which we define Suzuki, separate the shared core from the debugging harness, and name the bug-hunting fleet.

Part I · FoundationsCh. 1 · What Suzuki Is

Chapter One

What Suzuki Is

Suzuki is a self-improving debugging orchestrator. She receives a bug report, delegates bounded work to specialist minions, gathers executable evidence, proposes or applies minimal patches, verifies the fix, checks for regressions, and records wrapper lessons for future bug hunts.

1.1The two faces of the project

  • Suzuki the live fleet — one Letta orchestrator plus debugging minions that reproduce, inspect, test, patch, and verify.
  • The shared self-improvement core — the Mazda-style control plane that traces runs, judges results, proposes wrapper edits, A/B tests, gates, activates, and rolls back.
Definition 1.1 · Debugging Harness

A debugging harness is the replaceable domain layer that teaches the shared core how to work on code. It supplies repository adapters, command policies, test runners, static analyzers, failure-signature extractors, patch planners, regression verifiers, and debug evaluators.

1.2Siblings, not forks

Suzuki must not become a copied Mazda framework with finance words replaced by code words. Mazda and Suzuki should be two profiles over one kernel. The core stays domain-blind; finance and debugging are harnesses.

1.3What Suzuki optimizes

Table 1.1Debugging outcomes Suzuki should improve over time.
OutcomeMeaningEvidence
ReproducedThe bug can be triggered on demand.Failing command, failing test, stack trace, log excerpt.
LocalizedThe likely file/function/component is narrowed.Code references, call path, grep/blame/log evidence.
PatchedA minimal candidate fix exists.Git diff, rationale, changed tests.
VerifiedThe original failure is gone.Targeted test passes after failing before.
Regression-safeThe fix did not break nearby behavior.Relevant broader tests, lint, typecheck, smoke test.
LearnedThe wrapper improves for the next case.Approved prompt, memory, workflow, or tool-description patch.
Caution · No domain leakage into core

The common kernel must not import repository-specific, language-specific, Mazda-finance-specific, or Suzuki-debug-specific code. If a debugging concept is needed by the core, define a port. If it is Suzuki-specific, keep it behind the Suzuki harness factory.

Exercises

  1. (concept) Explain why Suzuki should be a harness swap rather than a framework fork.
  2. (design) Name three things that belong in the shared core and three that belong in suzuki_debug.
  3. (judgment) A teammate adds pytest imports to the kernel. Which boundary is broken?

System Topology: The Bug-Hunting Fleet

Suzuki follows Mazda's orchestrator-and-minion topology. The orchestrator owns the debugging plan and final report. Minions own narrow, repeatable slices of work through a shared Claude SDK executor and a debugging MCP/tool layer.

2.1The orchestrator memory blocks

The Suzuki orchestrator should render at least these system/ memory blocks: persona, human, debugging_procedure, tool_policy, team_agents, and repository_context.

Part II · The ArchitectureCh. 3 · Shared Core

Chapter Three

Shared Core and Domain Harnesses

The central task is to extract a Mazda-type self-improvement core, then hang domain harnesses off of it. This lets Suzuki inherit the hard parts: traces, wrapper revisions, experiments, gates, activation, and rollback.

3.1Target package boundary

agent_self_improvement/
  core/
    contracts/              # generic value objects and ports
    implementations/         # persistence, runtime, improvement, reporting
    runtime/                 # composition root, kernel, profile loading
    tests/                   # no finance, no debugging assumptions
  harnesses/
    mazda_finance/           # receipts, statements, finance evaluators
    suzuki_debug/            # repos, tests, stack traces, patches, regressions
  agent_packages/
    mazda/
    suzuki/

3.2What stays in the core

Table 3.1Shared-core responsibilities.
Core areaDomain-blind responsibility

FactoriesAbstract factories returning ports, not concrete finance/debug classes.
KernelRun task → persistrace → judge verdict → publish event.
PersistenceEvidence store for traces, verdicts, wrappers, experiments, approvals.
ImprovementProposal commands, experiment runner, activation, rollback.
ReportingGeneric run reports and dashboard primitives.

3.3What moves into Suzuki

Table 3.2Suzuki-spec responsibilities.
AreaExamples
Bug modelsBugReport, ReproductionCase, FailureSignature, PatchCandidate.
Repository adaptersGit status, diff summary, branch policy, file allow/deny lists.
Test adaptersPytest, npm, tsx, cargo, go test, project smoke tests.
Static-analysis adaptersTypecheckers, linters, dead-code scans, import-boundary checks.
Debug evaluatorsReproduction score, localization score, patch score, regression score.

3.4Interfaces to add before implementation

Keep the GoF hat on. If behavior varies by repository, language, test runner, execution environment, safety policy, or debugging strategy, make it a port before making it clever.

class IRepositoryWorkspace(ABC): ...
class ISourceControlAdapter(ABC): ...
class ICommandPolicy(ABC): ...
class ICommandExecutor(ABC): ...
class ITestRunner(ABC): ...
class IStaticAnalyzer(ABC): ...
class IFailureSignatureExtractor(ABC): ...
class IReproductionStrategy(ABC): ...
class IPatchPlanner(ABC): ...
class IPatchApplier(ABC): ...
class IRegressionVerifier(ABC): ...
class IDebugRunEvaluator(ABC): ...

Rule 3.1 · Interfaces before cleverness

Do not place a concrete debugging class directly in the kernel. First name the capability as a port, then write the adapter. This makes the first implementation slightly slower and every later harness swap much faster.

Part II · ArchitectureCh. 5 · The Loop

Chapter Five

The Bug-Finding Self-Improvement Loop

Suzuki's loop is Mazda's loop with a new evaluator and workflow. The stations remain run, judge, propose, experiment, gate, activate, and roll back. The evidence changes from receipts and totals to failing tests, stack traces, diffs, and regression checks.

5.1Trace fields

  • Bug report, repo, branch, and environment.
  • Commands run, exit codes, stdout/stderr tails, elapsed time.
  • Failure signatures: exception, stack frames, failing tests, log markers.
  • Files read/edited and git diff summary.
  • Targeted and broad test results before/after patch.
  • Minion delegation records and tool-call cost.

5.2Judge with deterministic evidence first

Principle 5.1 · Tests before taste

The first verdict source is executable evidence: failing test before, passing test after, and no new relevant regressions. LLM judgment may explain or classify; it does not overrule deterministic test evidence.

Table 5.1Suzuki scorecard dimensions.
DimensionGood evidenceBad evidence
ReproductionKnown failing command/test fails before patch.Vague diagnosis only.
LocalizationSpecific file/function tied to trace.Random broad guesswork.
Patch minimalitySmall diff targeting the fault.Large unrelated rewrite.
Fix verificationOriginal failing test passes after patch.Patch applied without rerun.
Regression safetyRelevant broader tests still pass.New failures or skipped checks.
EfficiencyFewer wasted commands/tokens over similar cases.Long trace with no evidence gain.

5.3Wrapper edits as commands

DebugPromptPatchCommand(
    target="suzuki-reproducer-agent/system/persona",
    reason="The agent inspected code before finding a failing command.",
    patch="Require a failing command or explicit non-reproducible verdict before localization."
)

DebugWorkflowPatchCommand(
    target="debug_reproduce_patch_verify",
    reason="Typecheck was skipped after TypeScript edits.",
    patch="After editing .ts/.tsx files, run npm typecheck when package.json exposes it."
)

5.4Gates

Table 5.2Suzuki promotion gates.
GateQuestion
CommandSafetyGateDoes the wrapper encourage dangerous commands, broad deletes, or secret exposure?
ReproductionGateDoes it preserve or improve finding a failing command first?
RegressionGateDid any previously solved bug task regress?
PatchQualityGateDoes it reduce patch bloat and unrelated changes?
CostGateDoes it stay within token/latency/tool-call budget?
UsefulnessGateIs the improvement material enough to activate?

Part III · PracticeCh. 6 · Migration

Chapter Six

Migration Plan: Mazda Core to Suzuki Harness

Do the migration by extraction, not rewrite. First prove what generic. Then move finance code behind a finance harness. Then build Suzuki's debugging harness against the same contracts. At each step, tests prove Mazda still works.

6.1Carve out the shared core

  1. Create agent_self_improvement/core and move generic contracts, runtime, persistence, improvement, and reporting there.
  2. Add import-boundary tests proving core imports no harness packages.
  3. Keep compatibility shims for old import paths until callers move.
  4. Run existing tests after each move.

6.2Move finance behind a harness

harnesses/mazda_finance/
  contracts/
    finance_models.py
    finance_ports.py
    finance_factory.py
  implementations/
    verifiers.py
    mcp_tools.py
    finance_evaluation_factory.py
  agent_package/
    mazda_package.py

6.3Create Suzuki in slices

Table 6.1Recommended implementation slices.
SliceGoalDone when
0Profile buildssuzuki_debug_default builds a kernel with stub adapters.
1Read-only triageSuzuki inspects files and summarizes likely fault zones without writing.
2ReproductionA failing command/test is captured as structured evidence.
3Patch candidateA minimal diff is produced behind write-root policy.
4VerificationTargeted test passes and relevant regression check runs.
5Self-improvementA wrapper edit is proposed, A/B tested, gated, approved, and activated.

6.4Tests to add early

test_core_import_boundaries.py       # core imports no harnesses
test_runtime_profiles.py             # Mazda and Suzuki profiles build via one composition root
test_debug_harness_factory.py        # factory creates workspace, source control, command policy, test runner
test_command_policy.py               # dangerous commands blocked or escalated
test_reproduction_evaluator.py       # failing command before patch scores higher than vague diagnosis
test_patch_quality_gate.py           # broad unrelated diffs fail the gate
test_regression_gate.py              # new failures block activation
Migration principle

Do not remove a Mazda finance implementation until a test proves the same core behavior still exists behind a harness boundary. The migration is complete when Mazda and Suzuki are two profiles over the same kernel.

Part III · PracticeCh. 7 · Build & Operate

Chapter Seven

Building, Testing, and Operating

Suzuki should be easy to run locally because debugging is local by nature. Her first useful environment is a checked-out repository, a safe command policy, a known test command, and a writable branch.

7.1Local commands

# all tests
../../.venv/bin/python -m pytest

# core only
../../.venv/bin/python -m pytest agent_self_improvement/core/tests

# Suzuki harness only
../../.venv/bin/python -m pytest agent_self_improvement/harnesses/suzuki_debug/tests

# read-only bug hunt
../../.venv/bin/python suzuki_run_bug_hunt.py   --repo /path/to/project   --bug-report bug_report.md   --mode read-only

# patch mode with approved write roots
../../.venv/bin/python suzuki_run_bug_hunt.py   --repo /path/to/project   --bug-report bug_report.md   --mode patch   --allowed-write-root src   --allowed-write-root tests

7.2Command policy defaults

Table 7.1Default safety policy.
Command classDefault behavior
Read-only inspection: ls, cat, grep, git statusAllowed in repo root.
Test commands: pytest, npm test, cargo testAllowed when configured in profile.
Package installs or network callsRequire approval unless sandbox profile allows them.
Destructive commands: rm -rf, reset/clean, broad chmod/chownBlocked or require explicit approval.
Secret-reading commands: env dumps, credential files, SSH keysBlocked by default.

7.3Operating the live fleet

curl -s -X POST http://<LETTA_HOST>/v1/agents/<SUZUKI_AGENT_ID>/recompile   | grep -oE '<projection>[^<]*</projection>'
# Expected Suzuki orchestrator count: 6 blocks.
# Expected minion count: 2 or more, depending on shared + specialty memory.
Caution · Do not trust a green report without reproduction

A debugging agent can look successful by writing a plausible explanation. Suzuki succeeds only when the trace conains executable evidence: a failing signal before the fix, a passing signal after the fix, and an appropriate regression check.

7.4Definition of done for Phase 1

  1. Core extracted and import-boundary tests passing.
  2. Mazda finance harness still runs through the shared core.
  3. Suzuki profile builds a kernel through the same composition root.
  4. Debug harness can run read-only bug triage and persist a trace.
  5. Reproduction evaluator scores at least one failing command/test.
  6. Command policy blocks destructive and secret-reading commands.
  7. HTML report shows bug report, reproduction, diagnosis, patch status, and next action.

Back MatterAppendix B

Appendix B

Glossary

Adapter. A concrete class satisfying a port while hiding a tool-specific API such as pytest, git, npm, or a linter.

Bug-hunting fleet. Suzuki plus her specialist Letta minions.

Command policy. The safety layer that decides whether a command is allowed, blocked, or requires approval.

Debugging harness. Suzuki's domain package: repositories, commands, tests, static analysis, patches, and regression verification.

Evidence store. The shared database of traces, verdicts, wrapper revisions, experiments, gates, approvals, and snapshots.

Failure signature. A normalized representation of a failure built from test names, exception text, stack frames, or log markers.

Harness swap. Replacing a domain implementation family while keeping the common self-improvement core intact.

Mazda-type agent. Any Letta agent built on the shared self-improvement framework: Mazda for finance, Suzuki for debugging, and future siblings.

Port. An abstract interface that callers use instead of a concrete implementation.

Regression verification. Comparing baseline and candidate behavior to ensure a patch did not introduce new failures.

Wrapper. The versioned, roll-back-able envelope around a fixed LLM: prompts, memory, tool descriptions, context rules, workflow steps, and minion templates.
