This manual sets one rule and explains why it holds: agent memory on this server is authored as files in a git repository (MemFS) and never again through raw POST/PATCH /v1/blocks calls. Memory blocks do not vanish — on this custom Docker deployment they remain the substrate the server projects into the system prompt — but they stop being a thing we write to directly. The git repository becomes the single source of truth; the attached blocks become a read-only projection of it. What follows is the principle, the evidence from Letta’s own roadmap, the architecture as it is versus as it must become, the exact write paths (with sequence diagrams), the repository conventions, a worked migration of the Mazda agent, the documentation we must rewrite, and the known hazards that will bite anyone who ignores them.

Chapter OneFirst Principles: What “Memory” Means Here

A Letta agent carries state between sessions. That state is what separates a stateful agent from a stateless chat. Historically that state lived in memory blocks: named chunks of text (persona, human, and custom labels) stored in the server’s database and spliced into the system prompt on every turn. The agent edited them through dedicated server tools (memory, core_memory_replace, memory_apply_patch).

MemFS (“memory filesystem,” also called a context repository) reframes that same state as a directory of markdown files under git. The agent edits files with ordinary bash tools and commits; git history becomes a complete, inspectable changelog of everything the agent has learned. Two categories of file exist, and the distinction is the whole efficiency argument:

CategoryPathLoading behaviorUse for
System filessystem/**Loaded fully into context every turnPersona, durable project facts, the team roster — things needed constantly
Other filesreference/**, notes/**, …Only the file tree + description frontmatter is shown; full text loads on demandHistory, handoffs, deep references — recalled only when relevant
The mental model Blocks are a database row view of memory. MemFS is a filesystem + git view of the same memory. The argument of this manual is that we should only ever touch the filesystem view, and let the server keep the row view in sync as a derived artifact.

Chapter TwoThe Official Position (and its Docker caveat)

Letta’s own documentation and roadmap are unambiguous about direction:

“MemFS replaces memory blocks for agents on the Letta API and in Local mode.” — MemFS, Letta Docs
“Legacy server memory tools like core_memory_replace will be removed in favor of straightforward filesystem operations on git-backed context repositories.” — Context Repositories, Letta Blog

New agents on the API and in Local mode have MemFS on by default; /memfs enable upgrades an existing agent by detaching the legacy memory(…) tool and migrating its blocks into the git filesystem. So far, exactly the directive we were given.

The caveat that makes this deployment special The same docs state: “MemFS is not available when using a Docker server — those agents continue using the legacy memory blocks system.” We run self-hosted Docker. By the stock product, we would be stuck on blocks. We are not — because this deployment ships a custom memfs. Chapter Three explains how, and why that changes what “phase out blocks” can mean here.

Chapter ThreeThis Deployment’s Reality

This server is a hand-built Letta stack. Memfs is implemented by a companion container, letta-memfs, plus a git-over-HTTP proxy exposed on the main API port:

  • Each memfs-enabled agent owns a bare repository reachable at localhost:8283/v1/git/<agent-id>/state.git.
  • Enablement is marked by a git-memory-enabled row in the Postgres agents_tags table (in the letta schema).
  • Bare repos live on the host at ~/.letta/memfs-repos/<agent-id>/state.git, owned root:root (this matters — see Chapter Eleven).

The decisive implementation detail: on this server, files committed under system/ are synced by the server into attached memory blocks. The block is how the text reaches the system prompt; the file is how we author it. That means blocks cannot be “removed” outright here — they are the projection surface. “Phase out blocks” therefore cannot mean delete all blocks; it must mean stop writing blocks directly and let memfs own them.

A contradiction we are deliberately overruling A note committed to Mazda’s own repo on 2026‑06‑13 (reference/history/handoff-2026-06-13-mazda-memfs-parity.md) asserts: “memory blocks are NOT being phased out here; memfs is an addition, not a replacement.” As of 2026‑06‑15 that guidance is superseded. Letta’s roadmap and this manual both treat blocks as a derived projection, not a write target. The old note is preserved for history but no longer governs.

Chapter FourThe Core Tension & Its Resolution

The whole problem reduces to one diagram’s worth of tension:

ForceSays
Letta roadmapBlocks are legacy; write memory as git files.
Docker caveatDocker servers “continue using blocks.”
This deploymentCustom memfs exists — but it is built on blocks (files project into blocks).
Today’s directive“Use memfs for all memory. Do not use blocks. If you see a block, fix it.”
Resolution (the governing policy)

1. Author memory only as MemFS files. All durable memory content is a markdown file committed to the agent’s state.git. Constant-context facts go in system/; everything else goes in reference/ et al.

2. Never call the block API directly. No POST /v1/blocks, no PATCH …/core-memory/blocks to create or edit memory content. Those calls are the “block” usage we are eliminating.

3. Blocks are a read projection. Attached system blocks may continue to exist because the server materializes them from system/ files. We treat them as output, not input.

4. When you find a hand-made block, reconcile it. Move its content into the repo (if not already there) and remove the raw block so the file is the only writer.

Chapter FiveArchitecture: As‑Is vs To‑Be

Figure 1 shows today’s hazard: two write paths terminate at the same attached blocks, and they can silently disagree. Mazda is living proof — her team_agents block was created by the raw API while an identical system/project/agent_team.md sits in her repo. Two writers, one surface, guaranteed drift.

Figure 2 is the target: one writer. The git repository is canonical. The server projects system/ files into blocks as a derived, read-only view. Path A is abolished.

Chapter SixThe Write Paths (Sequence Diagrams)

6.1 The deprecated path do not use

This is the anti-pattern we are retiring. It writes a block straight into Postgres and attaches it. It works, briefly — and it leaves the git repo stale, so the next person who reads the repo sees a different reality than the agent does. 

# ANTI-PATTERN — the kind of call this manual abolishes
curl -sL -X POST http://localhost:8283/v1/blocks/ \
  -H 'Content-Type: application/json' \
  -d '{"label":"team_agents","limit":5000,"value":"...table..."}'
curl -sL -X PATCH http://localhost:8283/v1/agents/<id>/core-memory/blocks/attach/<block_id>

6.2 The MemFS path use this

Author the same fact as a file, commit it, and let the server project it into a block. The repo stays authoritative; git records why the change happened. 

# THE PATTERN — author memory as a file, let memfs own the block
git clone http://localhost:8283/v1/git/<agent-id>/state.git mem && cd mem
$EDITOR system/project/agent_team.md            # create/update the fact
git add system/project/agent_team.md
git commit -m "team: correct minion roster"     # the "why" lives in history
git push origin main                            # server syncs system/** into the block

6.3 The ideal path: the agent edits its own memory target state

The end goal is that the agent never needs an operator for routine memory. It uses its own bash tools to edit a file and commit — exactly as Letta’s roadmap describes (“generalized computer-use tools like bash that operate over memory projected into git-backed files”).

Chapter SevenRepository Layout & File Conventions

A memfs repository follows a small, predictable shape. Adhere to it so loading behavior is predictable:

state.git  (bare, server-side)  ──clone──▶  working copy:
  system/                        # loaded FULLY every turn
    persona.md                  # who the agent is
    project/
      agent_team.md             # the minion roster (Mazda)
      decisions/memfs.md        # durable decisions
  reference/                     # loaded ON DEMAND (tree+description only)
    history/handoff-*.md
    vendors.md
  notes/                         # scratch, on demand

File format

Every file is markdown with YAML frontmatter. description is required (it is what the agent sees in the on-demand tree); limit is optional.

---
description: Mazda's helper-agent team (minions) and known Letta agents.
limit: 5000
---
# Mazda Helper Agents (minions)

| Purpose | Agent name | Agent ID |
|---|---|---|
| Parse / extract a statement | mazda-parser-agent | agent-a506… |
| …                            | …                  | …          |
Rule of thumb If a fact must be in the agent’s head on every turn, it belongs in system/. If it is something to look up when relevant, keep it out of system/ so it does not burn context every turn.

Chapter EightMigration: The Mazda Case Study

Mazda is the canonical worked example because she exhibits exactly the drift Figure 1 warns about. Her current attached blocks are: persona, human, environment, db_schema, verification_procedure, and the offending team_agents (block-3b051764…, created via raw API). Her repo already contains the equivalent system/project/agent_team.md. So the fix is reconciliation, not authoring.

Procedure (pending your go-ahead — touches a live agent)

  1. Confirm parity. Diff the raw block’s value against system/project/agent_team.md. They are reported identical; verify before deleting anything.
  2. Make the file canonical. If the block has any content the file lacks, fold it into the file first, commit, and push so nothing is lost.
  3. Detach & delete the raw block. Remove block-3b051764… from Mazda so the file is the sole writer. (If the server re-projects a team_agents block from system/, that projection is fine — it is derived, not hand-made.)
  4. Record the policy flip. Append a dated entry to system/project/decisions/memfs.md noting that as of 2026‑06‑15 raw block writes are deprecated in favor of file commits — superseding the 2026‑06‑13 note.
  5. Verify. Re-list Mazda’s blocks; confirm no hand-made block remains and the next turn still sees the roster.
GuardrailNo live-agent mutation happens without your explicit approval, per standing policy on Mazda and her minions.
Verification outcome — 2026‑06‑15: deletion HELD Step 1 parity passed (the file is a superset of the block). But step 3 cannot proceed safely yet: the team_agents block appears to be the sole live carrier of the roster. It is the only roster-bearing attached block, no separately-projected block exists, and there is no evidence the server injects memfs system/** into Mazda’s server-run runtime context. The GET /v1/agents/<id>/context endpoint is no help — it reports num_tokens_core_memory=0 and num_tokens_memory_filesystem=0 for every agent in this build (confirmed against Jeri, whose 73k-token context unquestionably carries her roster). The Chapter-8 assumption that “the server re-projects team_agents from system/” is therefore unproven on this deployment. Deleting the block now would risk dropping the roster from Mazda’s working memory. Resolution: keep the block until memfs→runtime injection for server-run agents is demonstrated (e.g. a memfs-aware session syncs her repo, or the server is shown to render system/** into context).

Chapter ElevenRisks & Known Bugs

11.1 The root:root bare-repo trap

The server-side bare repos at ~/.letta/memfs-repos/<id>/state.git are owned by root. A plain git push over the HTTP proxy is the normal path, but direct host-side pushes fail on permissions. The proven workaround is the bundle method: clone, edit, git bundle, copy the bundle into the letta-memfs container, and git fetch it as root (Appendix B).

11.2 The agents_tags duplicate-row bug resolved 2026-06-15

Status: fixedAs of 2026‑06‑15 this bug is resolved. A unique_agent_tag UNIQUE index on (agent_id, tag) is in place, the ~19k duplicate rows are gone, and the whole table holds 140 rows with zero duplicates. The history below is retained for context.

agents_tags historically had no unique constraint. Scissari had accumulated ~19,000 duplicate git-memory-enabled rows from some process inserting the tag unconditionally. This had to be fixed before leaning harder on memfs fleet-wide, or every memfs-enabled agent risked the same unbounded growth. 

-- Confirmed bug: no PK/unique index on agents_tags
SELECT agent_id, tag, count(*) FROM agents_tags
GROUP BY 1,2 HAVING count(*) > 1 ORDER BY 3 DESC;
-- Scissari → ~18,992 rows of the same (agent_id, 'git-memory-enabled')
-- Remediation (sketch): dedupe, then constrain, then fix the insert path
DELETE FROM agents_tags a USING agents_tags b
  WHERE a.ctid < b.ctid AND a.agent_id=b.agent_id AND a.tag=b.tag;
ALTER TABLE agents_tags ADD CONSTRAINT uq_agent_tag UNIQUE(agent_id, tag);
-- and make the writer use ON CONFLICT DO NOTHING

11.3 Sync conflicts & headless hangs

Memfs sync can conflict (two writers, or a stale clone) and, in headless contexts, can hang at “Checking for pending approvals…” — almost always SSH/key auth failing on a remote. The file-only policy reduces conflict surface (one writer), but operators must still use the Tailscale MagicDNS host for WSL remotes and verify ssh … echo ok before enabling a remote.

AppendixCommand & Bundle Reference

A. Everyday memfs commands

# Letta Code slash commands
/memfs enable            # migrate an agent's blocks into git-backed memfs
/memfs enable --local    # local-only (self-hosted) memfs
/memfs status            # show sync state
/memfs sync              # pull/push latest
/init                    # build hierarchical memory files after enabling

B. The bundle method for root:root bare repos

# When a direct push fails because the bare repo is root-owned:
git clone http://localhost:8283/v1/git/<id>/state.git mem && cd mem
# …edit files under system/ …
git add -A && git commit -m "mem: update"
git bundle create /tmp/mem.bundle main
docker cp /tmp/mem.bundle letta-memfs:/tmp/mem.bundle
docker exec -u root letta-memfs sh -c \
  'cd /repos/<id>/state.git && git fetch /tmp/mem.bundle main:main'
