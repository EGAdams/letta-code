What's actually wrong with these reports (root causes found so far)

  • Wrong statement copy-pasted wholesale. Jet Blue PDF 2's entire body (Account Reconciliation, Checks, Daily Balance, MyAdvance, Rewards, JSON) was byte-for-byte the Fifth Third checking account 5938 report — wrong bank, wrong account. Only the title and the transactions table were ever correct. Confirmed (by comparing hero/Source-PDF text across all reports) this was a one-off, not a tree-wide pattern — but still worth a quick sanity check on each remaining report.
  • Verified Transactions table never restructured. Jet Blue PDF 1 had all 10 contract sections and correct prose, but its table was never run through restructure_verified_transactions.py — no id="verified-transactions", no data-vendor-key, no onclick. The Set Category / Ask Mazda dialog markup existed at the bottom of the file but nothing on the page ever called it. This file was also the "known-good sibling" example cited in REPORT_OUTPUT_CONTRACT.md's Definition of Done — that reference is wrong and still needs fixing (see Task 11 below, not started).
  • Sign-direction bug recurring per-merchant. A small refund/credit from "RBT BP#9332735" was mislabeled as a positive charge in both JetBlue statements (PDF 1 and PDF 2) — same merchant, same bug, found independently in each. Always verify signs against the statement's own printed Balance Summary math (previous balance ± each line should equal the printed current/ending balance exactly); if it's off, you have a sign or total error.
  • Vendor keys were category names, not merchants. e.g. "travel_vehicle" instead of "mister_car_wash_0465". Found and fixed in both JetBlue reports.
  • Duplicate DB rows are everywhere. Nearly every transaction we checked across both JetBlue statements had 2–3 rows in expenses at the same (date, amount) — same real-world charge entered multiple times by different processes/agents over time. This matches a pattern already flagged in memory (rol_finances_llm_parser_credit_debit_bug_2026_06_18). Expect this in every remaining report — budget time for it.
  • Stale/wrong receipt_url values are common. Found multiple DB rows whose stored receipt path either doesn't exist on disk at all, or — worse — is a different transaction's receipt filename reused (e.g. a $33.13 receipt attached to a row claiming $26.83, or the same car-wash receipt reused across 3 different dates). Always verify a receipt_url with find ~/rol_finances/readable_documents/receipts -iname "<basename>" before trusting it. Clear bogus ones to NULL; never guess a replacement that doesn't match the row's exact (date, amount).
  • Found one fully corrupted DB row (Jet Blue PDF1, id 809): its id_light implied one date+amount, its actual amount column held a different value, and its receipt_url belonged to an entirely different, already-correctly-recorded transaction. Deleted it after confirming the real transaction it seemed to reference was already fine elsewhere. If a duplicate/insert collision looks like this, don't assume it's a normal duplicate — check carefully before deleting.
  • Missing DB rows. Jet Blue PDF1 had 2 of its 14 transactions completely absent from expenses — had to be INSERTed.

The correct workflow (don't hand-write the picker JS — use the real tool)

Early in this session, Jet Blue PDF 2 was fixed by hand-writing ~400 lines of picker boilerplate (it works, verified, but wastes effort). Jet Blue PDF 1 was fixed properly — use this approach going forward:

  1. Read ~/rol_finances/tools/python_tasks/verification_lib/REPORT_OUTPUT_CONTRACT.md first, every time.
  2. Read ~/rol_finances/readable_documents/bank_statements/january/business_january_february_6285/report.html in full as the structural model (10 required <h2> sections + an 11th "Verified Transactions" section). Don't edit this file — reference only.
  3. Extract real PDF text with PyPDF2, not pdfplumber (pdfplumber has misread statement PDFs before — wrong page count, dropped pages).
  4. Verify the report's own claims against the real PDF — watch for copy-paste-from-wrong-statement bugs.
  5. For every transaction, query expenses by (date, amount); insert if missing, dedupe if multiple, verify every receipt_url against the filesystem before trusting it.
  6. Resolve each row's category via REPORTING_CATEGORY_ANCESTOR_MAP in ~/rol_finances/receipt_parsing_tools/create_spreadsheet.py (parse with ast.literal_eval — don't import create_spreadsheet, xlsxwriter isn't installed in plain python3; walk categories.parent_id up until an id is in the map).
  7. Write the Verified Transactions table as a plain 6-column table with exact headers Date, Category, Vendor Key, Description, Signed Amount, Status (vendor key wrapped in <code class="inline-code">), one <tr class="cat-..."> per row. Do NOT add data-* attributes or onclick yourself.
  8. Run the real injector: python3 ~/rol_finances/tools/python_tasks/verification_lib/restructure_verified_transactions.py <report.html> — this auto-adds id="verified-transactions", the four data-* attributes, onclick, AND injects the entire category-picker dialog/CSS/JS in one idempotent step.
  9. Gotcha: if your source table cells contain a literal & written as &amp;, the injector's attribute-escaper double-escapes it into &amp;amp;. Fix after running the script with: sed -i 's/&amp;amp;/\&amp;/g' report.html (i.e. collapse the double-escape back to single), then re-grep to confirm.
  10. Don't hardcode data-receipt-url/has-receipt — the page already resolves receipts live via /api/receipts-present on load.
  11. Verify: grep -o 'data-vendor-key=' file | wc -l equals row count (repeat for the other 3 data-* attrs), grep -c 'rol-category-picker:start' file is 1, HTML is well-formed, and curl -s -o /dev/null -w '%{http_code}\n' http://localhost:8765/rol_finances_reports/<dir>/report.html returns 200.
